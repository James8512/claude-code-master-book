## 개요

Claude Code에는 AI에게 지식과 규칙을 전달하는 여러 메커니즘이 있습니다. CLAUDE.md, Skills, Slash Commands, Plugins는 각각 다른 목적과 특성을 가지며, 상황에 따라 적절한 도구를 선택해야 합니다.

## 비교 테이블

| 특성 | CLAUDE.md | Skills | Slash Commands | Plugins |
|------|-----------|--------|----------------|---------|
| **로딩 시점** | 세션 시작 시 자동 | 관련 작업 감지 시 자동 | 사용자가 명시적 호출 | 설치 시 등록 |
| **주요 사용자** | 프로젝트 전체 팀 | 특정 역할/작업 전문가 | 반복 작업 수행자 | 기능 배포자 |
| **토큰 효율성** | 항상 소비 (상시 로드) | 필요 시에만 소비 | 호출 시에만 소비 | 구성 요소에 따라 다름 |
| **범위** | 프로젝트 전체 | 특정 작업 도메인 | 단일 명령 | 기능 번들 |
| **관리 위치** | 프로젝트 루트 | ~/.claude/skills/ | 슬래시로 호출 | /plugin 명령어 |
| **공유 방식** | Git으로 공유 | 파일 복사 | Plugin에 포함 | npm/GitHub |

## 각 메커니즘의 적합한 사용 사례

### CLAUDE.md

- 프로젝트의 기술 스택과 아키텍처 정보
- 팀 전체가 따라야 하는 코딩 규칙
- 절대로 해서는 안 되는 금지 사항
- 빌드/배포 명령어

<Callout type="warning" title="CLAUDE.md 비대화 주의">
CLAUDE.md에 모든 것을 넣으면 매 세션마다 토큰을 소비합니다. 항상 필요한 정보만 CLAUDE.md에, 특정 작업에만 필요한 정보는 Skills에 넣으세요.
</Callout>

### Skills

- 코드 리뷰 체크리스트
- TDD 워크플로우 가이드
- 특정 API 사용법
- 번역/문서화 규칙

### Slash Commands

- 자주 반복하는 워크플로우 시작
- 컨텍스트 핸드오프
- 프로젝트 초기화 작업

### Plugins

- 여러 프로젝트에서 재사용할 기능 번들
- 팀이나 커뮤니티에 배포할 도구 모음
- Hooks + Skills + MCP가 함께 동작해야 하는 기능

## ykdojo의 조언: 수렴의 방향

<Callout type="info" title="ykdojo의 관점">
"이 네 가지 메커니즘은 점점 수렴하고 있습니다. 미래에는 더 통합된 방식이 나올 수 있지만, 현재로서는 각각의 강점을 이해하고 조합하는 것이 중요합니다. **간단한 것은 CLAUDE.md에, 전문적인 것은 Skills에, 반복적인 것은 Commands에, 공유할 것은 Plugin에** 넣으세요."
</Callout>

## 의사결정 플로우

<Steps>
<Step title="항상 필요한 정보인가?">
Yes → CLAUDE.md에 추가
</Step>
<Step title="특정 작업에만 필요한 전문 지식인가?">
Yes → Skill로 작성
</Step>
<Step title="자주 반복하는 명령 시퀀스인가?">
Yes → Slash Command로 등록
</Step>
<Step title="여러 기능을 묶어서 공유해야 하는가?">
Yes → Plugin으로 번들링
</Step>
</Steps>
