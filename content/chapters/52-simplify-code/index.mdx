## 복잡한 코드를 단순하게

소프트웨어가 성장할수록 불필요한 복잡성이 쌓입니다. 과도한 추상화, 사용되지 않는 패턴, 역사적 이유로 남아있는 레이어들이 코드를 읽기 어렵게 만듭니다. Claude Code는 이런 복잡성을 객관적으로 분석하고 제거하는 데 탁월합니다.

<Callout type="tip" title="ykdojo의 조언">
"충분히 질문하면 오히려 더 빠르게 이해할 수 있습니다." Claude Code에게 코드의 복잡성에 대해 계속 질문하세요. "왜 이렇게 되어 있는지", "더 단순한 방법은 없는지" 반복해서 물으면, AI가 점점 더 핵심에 가까운 답을 제시합니다.
</Callout>

## 4단계 코드 단순화 프로세스

<Steps>
<Step title="복잡성 설명 요청">
먼저 Claude Code에게 코드의 현재 구조를 설명하게 합니다.

```
> 이 모듈의 전체 구조를 설명해줘.
  각 클래스와 함수의 역할, 의존 관계, 데이터 흐름을 정리해줘
```

이 단계에서 코드의 전체 그림을 파악합니다. Claude Code가 코드를 "읽고 이해하는" 과정 자체가 구조적 문제를 드러냅니다.
</Step>
<Step title="불필요한 추상화 식별">
```
> 이 코드에서 불필요한 추상화나 과도한 복잡성이 있는 부분을 찾아줘.
  다음 기준으로 분석해줘:
  - 한 번만 사용되는 인터페이스나 추상 클래스
  - 실제로는 하나의 구현만 있는 팩토리 패턴
  - 단순히 다른 함수를 호출하기만 하는 래퍼 함수
  - 실질적 기능 없이 데이터를 전달하기만 하는 레이어
```

Claude Code가 객관적으로 코드를 분석하여 "이 부분은 불필요합니다"라고 지적합니다.
</Step>
<Step title="단순한 버전 제안">
```
> 불필요한 추상화를 제거하고, 같은 기능을 하는 더 단순한 버전을 제안해줘.
  코드 라인 수와 파일 수가 줄어들어야 해
```

Claude Code가 리팩토링된 버전을 제안합니다. 종종 코드 라인 수가 절반 이하로 줄어드는 경우도 있습니다.
</Step>
<Step title="장단점 비교">
```
> 현재 버전과 단순화된 버전의 장단점을 표로 비교해줘.
  가독성, 유지보수성, 확장성, 테스트 용이성 기준으로
```

| 기준 | 현재 버전 | 단순화 버전 |
|------|-----------|------------|
| 가독성 | 여러 파일을 오가야 함 | 핵심 로직이 한눈에 보임 |
| 유지보수성 | 변경 시 여러 레이어 수정 | 한 곳만 수정하면 됨 |
| 확장성 | 이론적으로 유연 | 실제 필요할 때 확장 가능 |
| 테스트 용이성 | 모킹이 복잡 | 직접 테스트 가능 |
</Step>
</Steps>

## 실전 사례: 과도한 추상화 제거

### Before: 5개 파일, 150줄

```
UserController → UserService → UserRepository → UserMapper → UserEntity
```

각 레이어가 거의 아무런 로직 없이 다음 레이어를 호출하기만 합니다.

### After: 2개 파일, 60줄

```
UserHandler → Database
```

실제 비즈니스 로직만 남기고, 불필요한 중간 레이어를 제거합니다.

```
> 이 UserController → UserService → UserRepository 체인에서
  각 레이어가 실제로 하는 일을 분석해줘.
  단순히 다음 레이어를 호출하는 것 이상의 로직이 있는 레이어만 남기고
  나머지는 합쳐줘
```

<Callout type="warning" title="주의: 무조건 줄이는 것이 아닙니다">
단순화가 항상 정답은 아닙니다. 팀의 컨벤션, 미래 확장 계획, 테스트 전략 등을 고려해야 합니다. Claude Code에게 장단점 비교를 요청하고, 팀과 논의한 후 결정하세요.
</Callout>

## 질문의 힘

코드를 단순화하는 가장 강력한 도구는 "질문"입니다.

- "이 인터페이스를 구현하는 클래스가 몇 개야?"
- "이 팩토리가 실제로 다른 객체를 생성하는 경우가 있어?"
- "이 래퍼 함수를 제거하고 직접 호출하면 어떤 문제가 생겨?"
- "이 디자인 패턴이 여기서 정말 필요해?"

충분히 질문하면, 코드의 본질이 드러나고, 무엇이 정말 필요하고 무엇이 불필요한지가 명확해집니다.
