## 왜 검증이 필요한가?

Claude Code는 강력하지만, 그 출력을 무조건 신뢰하면 안 됩니다. AI가 작성한 코드는 겉보기에는 완벽해 보여도 미묘한 논리 오류, 엣지 케이스 누락, 성능 문제를 포함할 수 있습니다. 체계적인 검증 과정이 필수입니다.

## 4가지 검증 방법

### 1. 테스트 코드로 검증

가장 객관적인 검증 방법입니다. Claude Code에게 코드와 함께 테스트를 작성하게 하고, 테스트가 실제로 통과하는지 확인합니다.

```
> 방금 작성한 함수에 대한 단위 테스트를 작성하고 실행해줘.
  엣지 케이스도 포함해줘
```

<Callout type="info" title="테스트의 한계">
테스트가 통과한다고 해서 코드가 완벽한 것은 아닙니다. 테스트 자체가 잘못될 수 있습니다. 특히 AI가 코드와 테스트를 모두 작성한 경우, 같은 오해가 양쪽에 반영될 수 있습니다.
</Callout>

### 2. GitHub Desktop 시각적 리뷰

코드 변경 사항을 GitHub Desktop이나 `git diff`로 시각적으로 확인합니다. 파일별로 변경 내용을 검토하면 예상치 못한 수정이나 삭제를 발견할 수 있습니다.

```bash
$ git diff --stat   # 변경된 파일 목록
$ git diff          # 상세 변경 내용
```

의도하지 않은 파일이 수정되었거나, 기존 코드가 삭제되었거나, 불필요한 변경이 포함되지 않았는지 눈으로 확인합니다.

### 3. 드래프트 PR 활용

변경 사항을 드래프트 PR로 올려서 팀원의 리뷰를 받습니다. AI가 작성한 코드라는 것을 알리면, 더 꼼꼼한 리뷰를 받을 수 있습니다.

```
> 현재 변경 사항으로 드래프트 PR을 만들어줘.
  PR 설명에 AI가 작성한 코드라는 점과 특히 리뷰받고 싶은 부분을 명시해줘
```

### 4. 자기 검증 (Self-Verification)

Claude Code에게 자신의 작업을 다시 검증하라고 요청합니다. 놀랍게도 이 방법이 상당히 효과적입니다.

```
> 모든 것을 다시 확인하고 검증 표를 만들어줘
```

이 프롬프트를 사용하면 Claude Code가 자신이 작성한 코드를 처음부터 다시 점검하고, 각 항목의 검증 결과를 표로 정리합니다.

| 검증 항목 | 상태 | 비고 |
|-----------|------|------|
| 타입 안전성 | 통과 | 모든 함수에 타입 정의됨 |
| 에러 처리 | 주의 | catch 블록에서 에러 로깅 누락 |
| 엣지 케이스 | 통과 | null, undefined, 빈 배열 처리됨 |
| 성능 | 통과 | O(n) 복잡도, 메모이제이션 적용 |

<Callout type="tip" title="검증 프롬프트의 힘">
"모든 것을 다시 확인하고 검증 표를 만들어줘"라는 단순한 프롬프트가 강력한 이유는, Claude Code가 "작성 모드"에서 "검증 모드"로 전환되어 비판적 시각으로 코드를 재검토하기 때문입니다.
</Callout>

## 검증 체크리스트

<Steps>
<Step title="기능 검증">
요청한 기능이 모두 구현되었는지 확인합니다. 누락된 요구사항이 없는지 하나씩 대조합니다.
</Step>
<Step title="회귀 검증">
기존 기능이 깨지지 않았는지 확인합니다. 기존 테스트가 모두 통과하는지 실행합니다.
</Step>
<Step title="보안 검증">
입력값 검증, 인증/인가, SQL 인젝션, XSS 등 보안 취약점이 없는지 점검합니다.
</Step>
<Step title="성능 검증">
불필요한 반복, 메모리 누수, N+1 쿼리 등 성능 문제가 없는지 확인합니다.
</Step>
</Steps>
