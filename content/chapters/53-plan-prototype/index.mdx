## 계획 먼저, 코드는 나중에

에이전틱 개발에서 가장 흔한 실수는 바로 코딩부터 시작하는 것입니다. Claude Code에게 코드를 작성하라고 하기 전에, 먼저 전체 그림을 그려야 합니다.

```
먼저 전체 계획을 세워줘.
사용할 기술 스택, 파일 구조, 각 기능의 위치를 결정해줘.
코드는 아직 작성하지 마.
```

이 접근 방식은 Claude가 전체 아키텍처를 먼저 사고하게 만들어, 나중에 발생할 수 있는 구조적 문제를 사전에 방지합니다.

<Callout type="tip" title="계획 프롬프트의 핵심">
"코드는 아직 작성하지 마"라는 문구가 중요합니다. 이것이 없으면 Claude는 즉시 코드를 생성하기 시작합니다. 계획 단계에서는 명확하게 코드 작성을 제한하세요.
</Callout>

## 빠른 프로토타입으로 비교하기

계획을 세운 후에는 최적의 접근 방식을 찾아야 합니다. 이때 유용한 패턴이 빠른 프로토타입 비교입니다.

```
두 가지 접근 방식을 빠르게 프로토타입으로 만들어서 비교해줘.
각각의 장단점을 분석하고, 어떤 것이 더 적합한지 추천해줘.
```

## 실제 사례: ykdojo의 Diff 뷰어

ykdojo는 Git diff 뷰어를 만들 때 이 패턴을 완벽하게 보여줬습니다. 처음에는 bash + tmux + lazygit 조합으로 시도했지만, 원하는 수준의 UI를 구현하기 어려웠습니다.

<Steps>
<Step title="첫 번째 프로토타입: bash + tmux + lazygit">
터미널 기반 도구들을 조합하여 diff 뷰어를 구현하려 했습니다. 빠르게 시작할 수 있었지만 커스터마이징에 한계가 있었습니다.
</Step>
<Step title="두 번째 프로토타입: Ink + Node.js">
React의 터미널 렌더링 라이브러리인 Ink를 사용하여 완전히 다른 접근 방식으로 프로토타입을 만들었습니다. UI 커스터마이징이 자유롭고 원하는 UX를 정확히 구현할 수 있었습니다.
</Step>
<Step title="비교 및 최종 선택">
두 프로토타입을 나란히 비교한 결과, Ink + Node.js가 더 유연하고 확장 가능한 솔루션임을 확인하고 이를 최종 선택했습니다.
</Step>
</Steps>

<Callout type="info" title="프로토타입의 가치">
프로토타입을 만드는 데 소요된 시간은 잘못된 아키텍처로 전체 프로젝트를 진행한 후 다시 시작하는 것보다 훨씬 적습니다. "두 시간의 프로토타이핑이 두 주의 리팩토링을 절약합니다."
</Callout>

## 계획과 프로토타입의 균형

| 상황 | 접근 방식 |
|------|-----------|
| 기술 스택이 확실할 때 | 계획 후 바로 구현 |
| 여러 접근 방식이 가능할 때 | 프로토타입 비교 후 선택 |
| 완전히 새로운 영역일 때 | 최소한의 계획 → 탐색적 프로토타입 → 본격 계획 |
| 기존 코드를 리팩토링할 때 | 현재 구조 분석 → 계획 → 점진적 구현 |
